# 大模型多节点 LoRA + DeepSpeed 训练性能调优备忘

## 1. 核心认知：系统工程的叠加效应

多节点 LoRA 的性能提升，从来不是单一靠修改某个参数（比如 `bucket_size`）就能实现质变的，拿 32B 模型在 24GB 显卡（如 RTX 3090/4090）上微调为例，这是一个**几项优化叠加的系统工程**：

1. **显存驻留（生存基底）**：首先利用 DeepSpeed 的 ZeRO 分片能力（ZeRO-2 甚至 ZeRO-3 及 Offload），解决 32B 庞然大物在残缺算力（24GB 显存）上的 OOM 驻留问题。活下来是第一步。
2. **减少跨节点通信（拓扑与 Batch 优化）**：调整并行策略（TP 通常局限于切分在节点内的高速 NVLink，跨节点则老老实实走 DP 配合 ZeRO），并优化 Batch 组织，从拓扑物理层面直接减少不跨机不行的大团通信。
3. **极致 Overlap（隐藏通信消耗）**：精细调节 Bucket Size、重叠通信（Overlap）、以及梯度累加（Grad Accumulation）的节奏，让**必须要做的通信**统统塞进 Backward 计算的影子里面去。
4. **流水线压榨（榨干空闲）**：最后优化数据输入端（如 Sequence Packing 和基于 Dataloader 的预取机制），把 GPU 等 DataLoader 数据喂入的时长、以及最后残存的等网络通信的时长，全军压境般榨干。

---

## 2. 核心矛盾：为什么 LoRA 不能照搬 Full FT 的 Bucket 设置？

在通信与计算资源协同调度的基本框架里，我们要特别搞懂 DeepSpeed 的 **Bucket（通信桶）** 机制。系统不会每个参数一算完梯度就通信，而是攒进 Bucket，满载后整体起一次 All-Reduce 或 Reduce-Scatter 通信。

* **在全量微调（Full FT）中**：几十亿参数都在产生梯度。算好的梯度瞬间就能填塞满一个 500MB 的默认大 Bucket（`reduce_bucket_size` / `allgather_bucket_size`），触发后台通信。于是，整条 Backward 时间线上，计算和通信处于极度健康的**并发交错（Overlap）**状态。
* **在 LoRA 中**：仅有注入的低秩矩阵 A/B 会产生梯度，总传输量仅有十几 MB 到百兆。
  * 若沿用 Full FT 的 500MB 大 Bucket，**整个 Backward 走到尽头，所有的 LoRA 梯度都填不满这一个 Bucket 的坑**！
  * **后果：** 没有任何后台通信会被提前触发并发走。系统生生把所有的网络通信全都憋到了 Backward 彻底算完的那一刻，集中释放。完全丧失了 Overlap 的优势。

---

## 3. Bucket 大与小的权衡 (Bandwidth vs. Latency)

* **Bucket 调大（向绝对带宽妥协）**：
  * **优势**：单次传输数据包大，能更容易跑满网卡极限带宽，总吞吐聚合猛。
  * **劣势**：**发车太晚**。因为需要等几十层的反向传播拼命填坑才能发车，完全错失了先处理好前面层的梯度的重叠机会。
* **Bucket 调小（向低延迟和高并发妥协）**：
  * **优势**：极容易填满并早早触发通信，疯狂 Overlap。算完尾部两三层的 LoRA 梯度就立马在后台小包发走，等整个 Backward 回到头部时，很多通信早就在暗中度陈仓完成了。
  * **劣势**：**消息过于细碎。** 集群跨机通信本身带有极高的启动过路费（RTT 网络时延 + NCCL Kernel Launch Overhead）。如果切得太碎，网卡一直处于在跑起步油门的状态，有效带宽极低，反而会被通信的高响应耗时拖挂。

所以，我们的调参关键点在于：**Step Time 看的不是你的单次 All-Reduce 把网卡拉得有多满，看的是那些被毫无掩蔽地“暴露在计算外的通信（Exposed Comm）”有多长。** 掩护在计算之下的低带宽通信也是好通信，暴露在外的高带宽通信也是差通信。

## 4. 寻找最优解的“强相关”调节参数

去探测完美平衡两者的那个关键阈值（“发车点”），你需要关注以下变量的耦合：

1. **LoRA Rank (`r`)**：Rank 越大，模型梯度体积略微见长，你可以适度扩大 Bucket 追求带宽。
2. **Trainable Params 散布结构**：如果你只对 Attention 的 `q_proj` 和 `v_proj` 下手，梯度的产生极度“稀疏”，Bucket 应当收窄才能更快攒满。若你对所有 Linear 全覆盖，梯度稍密集，Bucket 策略可稍放开。
3. **ZeRO Stage 的不同通信图**：
   * ZeRO-1/2 下主要管发货用的 `reduce_bucket_size`。
   * ZeRO-3 会涉及到参数重新拉取的全收集过程，必须留意配置对应的 `allgather_bucket_size`。
4. **Gradient Accumulation**：在梯度累加中间步，不需要跨机规约，不耗费大量网络。通信洪峰永远出现在 `micro_step == gradient_accumulation_steps` 的最后那一整个反向传播大波峰里。不要在此处做错误归因。
5. **跨机基建网络延迟**：低速以太网集群，用大点的 Bucket 抗 Latency；高端 IB 集群，用极碎的 Bucket 吃尽高频响应极光。
6. **Backward 梯度 Ready 时序**：特殊 Checkpointing 图会让参数倒序准备的时机变幻莫测。

## 5. Profiler 诊断视角：如何从现象直击本质？

排查问题的杀器是分析 PyTorch Profiler / Nsight Systems 等工具生成的 Timeline（时间刻度尺）。

* **理想情况（健康无忧）**：Backward 的计算执行流（CUDA Kernels）底部由于交错启动的 NCCL 流严丝合缝地重叠咬合。Backward 结束时，NCCL 只露出一丁点残余的尾巴，随即快速进入 Optimizer Step。
* **典型的 LoRA 失控灾难**：在监控图表里，你能清晰看到大片 Backward 计算早早退场，而由于前面的通信全被淤积（Bucket 太大未发车），在计算后面傻乎乎地**排队长达数时毫秒的裸露 NCCL 通信长尾**。这时候，最昂贵的 GPU 阵列犹如在等外卖一样纯空转！
* **对策**：看到尾部超长脱节通信流，立刻切入 DeepSpeed 配置文件削减 Bucket 尺寸，且关注计算死锁导致的时机错位。

## 6. 实战起跑线预置：GA 与 Bucket 经验法则

并没有一个放之四海而皆准的“绝对标准值”，但针对多节点 LoRA 场景，起手式是有规律可循的。
我们主要关注两个控制阀门：**Gradient Accumulation（梯度累加步数，GA）**和 **Bucket Size（通信桶大小）**。

### 6.1 Gradient Accumulation (GA) 策略
* **误区**：官方虽然经常默认 GA=1，但在跨机分布式中这是引发网络风暴的灾难。`有效 batch = per_device_train_batch_size × num_devices × GA`，系统会基于这个层级组织通信。多机下的“朴素”无 GA 同步往往伴随极其惨烈的通信拖慢 (Slowdown)。
* **起手式**：**永远不要从 1 开始，先试 4 或 8。**
  * 提高 GA 相当于放大了每次同步时所护航的“有效计算工作量”，让单次大规模通信变得物有所值。
  * **切忌一上来就拉满（如 32）**：超大 GA 导致 Effective Batch 剧增，虽然硬件利用率高了，但吞吐不一定能提升，且会大幅改变模型的收敛轨迹，逼迫你重新寻找学习率。
  * **诊断动作**：如果发现每卡的 Micro-batch 极小，且通信裸露明显，果断按 `4 -> 8 -> 16` 的阶梯扫一轮。

### 6.2 Bucket Size 策略
依据底层通讯后端不同，单位会有变化，但心法一致。
* **如果是纯 PyTorch DDP**：
  * 原生的 `bucket_cap_mb` 默认是 25 MiB。
  * **起手式**：顺着 `25 -> 50 -> 100 MiB` 的阶梯往上测。多节点 LoRA 千万别因为害怕细碎就上来设个巨型 Bucket，晚启动的倒逼效应同样极具破坏力。
* **如果是 DeepSpeed ZeRO**：
  * **不要盲信默认设定**。官方手册及许多祖传配置中的 `reduce_bucket_size` 默认是 `5e8 elements`（约等于 BF16/FP16 下的 954 MiB，FP32 下近 1.9 GB），这是留给全参数微调这头巨兽的标配。
  * **起手式**：针对 LoRA 轻盈的体态，**改从 `5e7 / 1e8 / 2e8 elements` 作为核心测试梯队**。

## 7. “黄金搭配”与 Profiler 调优指针

一个在业界经历过千锤百炼、远比无大脑“GA=1 + 巨大 Bucket”稳健的默认基线组合是：

> **`GA = 8`**  搭配  **`DDP Bucket = 50 MiB`** (或 **`ZeRO Bucket = 1e8 elements`**)

这套基准之所以稳若磐石，是因为它精准命中了两个平衡点：一方面通过 GA=8 垫厚了局部算力去掩盖通信；另一方面利用不碎也不晚的中型 Bucket 让暗流数据传递尤为平顺。

### 调优指针 (Troubleshooting)
看完了 Profiler 之后，你的旋钮应该往哪边转？

1. **满屏都是很碎的极小 NCCL collective 交织？**
   * 说明网卡被频繁的 Kernel Launch 和网络 RTT 延迟彻底拖跨。
   * **动作**：优先**调大 Bucket**，同时也可将 **GA 从 4 提向 8**。
2. **Backward 尾部拖着一条又粗又长的通信大尾巴？**
   * 直观反映就是“攒货太多、发车太迟”。此时千万不能继续加码。
   * **动作**：果断**缩小 Bucket** 让它提前发车。
3. **每张卡分到的 Token 数量极少，算力吃不饱？**
   * 此时单靠调通信 Bucket 已经无力回天。
   * **动作**：**优先将 GA 提升至 8 或 16**，或是直接从源头扩大每卡微批次的 Token 切片数，以此抢回算力主权。
